# Marketing Copywriting for Product Surfaces

Use this reference when writing or restructuring copy for software products across marketing sites, landing pages, feature pages, onboarding, upgrade prompts, lifecycle messages, app-store listings, and other conversion-sensitive surfaces.

This library is not trying to turn every design task into a growth-hacker carnival. The goal is clear, credible, action-oriented product copy that helps the right user understand the value and take the next step.

## Ask Only for Missing Context

Before writing, gather only what the prompt or workspace does not already make clear.

### 1. Surface and goal

- What surface is this for?
  - marketing site: homepage, landing page, feature page, comparison, about
  - product UX: onboarding, empty state, tooltip, permission prompt, upgrade prompt, settings
  - lifecycle: email, push, in-app message, release note
  - app store: listing text, captions, subtitle
- What is the **one** primary action?
  - install
  - start trial
  - create account
  - request access
  - book demo

### 2. Audience

- Who is the ideal user?
- What triggered the search for a solution now?
- What problem are they trying to solve?
- What objections matter most: switching cost, learning curve, trust, reliability, integration, privacy, compliance?
- What words do users already use to describe the problem?

### 3. Product and positioning

- What category is this in and what does it replace or compete with?
- What is the key transformation or outcome?
- What is the first meaningful win in the first 5-10 minutes?
- What are the most concrete 1-3 differentiators?
- What real proof exists: metrics, quotes, logos, case studies, ratings, awards?

### 4. Traffic and context

- Where is traffic coming from: ads, organic, referral, marketplace, app-store search, existing product?
- How warm is the visitor: cold, aware, comparison-stage, already evaluating?
- Mobile or desktop? Language or regional constraints?

### 5. Constraints

- Brand voice rules or phrases to avoid
- Claims that cannot be made
- Legal/compliance requirements

If the most important positioning inputs are unknown, say so. Strong copy depends on a real offer, real audience, and real proof.

## Core Principles

### Clarity over cleverness

Software already asks users to think hard. Copy should reduce interpretation work, not add a second puzzle.

### Outcomes over features

- **feature**: what it does
- **outcome**: what changes for the user

Translate features into practical results. `Sync calendars` matters because it prevents scheduling conflicts and manual updates.

### Specificity over vagueness

Prefer concrete examples, timeframes, and realistic scenarios over foggy claims.

### Customer language over internal language

Mirror how users describe the problem. Avoid product-marketing fog like `platform`, `innovative`, or `streamline` unless the term truly clarifies something.

### One idea per section, one job per screen

Especially on mobile, each screen should answer one question and support one next step.

### Proof near the claim

Do not ask for trust from thin air. Put evidence close to the promise it supports.

## Writing Style Rules for Software Products

Prefer:

- simple over complex
- concrete over buzzwordy
- active over passive
- confident over hedged
- honest over hyped

### Good defaults

- `use` over `utilize`
- `help` over `facilitate`
- `set up` over `implement`
- `smooth` over `seamless`
- `modern` over `cutting-edge`

### Avoid unless provable

- `never`
- `always`
- `guaranteed`
- `best`
- `leading`
- `trusted by thousands`

### Mobile microcopy rules

- keep it short and scannable
- lead with the action or benefit
- keep buttons to 2-4 words when possible
- for permissions, explain the benefit first, then ask
- for errors, say what happened, why, and how to fix it

## Quick Quality Check

Before shipping copy, ask:

- Is there jargon an outsider would miss?
- Is any sentence trying to do too much?
- Is there passive voice where active voice would be clearer?
- Is there a claim without proof nearby?
- Is this `we`-focused where it should be `you`-focused?
- Are exclamation points doing emotional labor the copy did not earn?

## Practical Moves

### Be direct

Lead with the value and the next step.

### Use trigger-based framing

Start from the moment the user feels the problem.

- `When release notes turn into archaeology...`
- `When customer questions pile up in Slack...`

### Use before → after contrast

Show the current mess, then the better state. Keep it concrete, not melodramatic.

### Use rhetorical questions sparingly

Only when the question matches what the reader is already thinking.

### Use analogies only when they simplify

If the analogy adds interpretation work, cut it.

### Use humor carefully

Only when it fits the brand and does not reduce clarity, trust, or seriousness.

## Structure Frameworks by Surface

### Marketing pages

#### Above the fold

- **headline**: the primary outcome for a specific user
- **subheadline**: who it is for, what it replaces, what happens next
- **primary CTA**: concrete and action-oriented

Useful headline frames:

- `{achieve outcome} without {pain point}`
- `The {category} for {audience}`
- `Stop {painful task} in {timeframe}`
- `{question that names the pain}`

#### Common section order

| Section | Job |
| --- | --- |
| Social proof | Build credibility early |
| Problem / pain | Show understanding of the stakes |
| Outcomes / benefits | Show what improves and why it matters |
| How it works | Reduce mystery in 3-4 steps |
| Use cases | Help readers self-identify |
| Integrations / compatibility | Reduce fit anxiety |
| Objection handling | Address trust, migration, or pricing concerns |
| Final CTA | Restate value and ask clearly |

### Product onboarding

- promise what the user will achieve in the next 2-5 minutes
- guide them to a first real win fast
- personalize only what improves results immediately
- ask for permissions just in time, not as a trust tax upfront
- invite sharing or collaboration only after value is felt

### Product UX microcopy

Applies to empty states, tooltips, upgrade prompts, settings, confirmations, errors, and helper text.

Keep the structure simple:

- what this is
- why it matters
- what to do next

Useful patterns:

- **empty state**: `No {items} yet` + value + CTA
- **tooltip**: one sentence, benefit-focused
- **confirmation**: what changed + what happens next
- **error**: what happened + likely cause + recovery step

### App-store listings

- lead with the strongest outcome in the first lines
- use bullets as outcomes, not feature inventory
- make the use cases easy to picture
- be clear about what the product is and is not

### Lifecycle messages

- one purpose per message
- subject line or notification lead should match the body promise
- one clear next step
- avoid spammy urgency or faux-importance

## CTA Guidance

Weak CTAs usually hide the action behind generic language.

Avoid when a stronger alternative is available:

- `Submit`
- `Learn more`
- `Click here`
- `Get started`

Prefer:

- `Install the app`
- `Start free trial`
- `Create your first workspace`
- `Import my data`
- `Try demo workspace`
- `Request access`
- `Book a demo`

### CTA formula

`{action verb} + {immediate value} + {qualifier when needed}`

Examples:

- `Create my first board`
- `Import a project`
- `Install the iOS app`
- `Request early access`

## Surface-Specific Notes

### Homepage

- serve multiple intents without becoming generic
- offer clear paths to evaluate, explore, trust, and start
- prove credibility early

### Landing page

- match the traffic source promise closely
- keep one message and one primary CTA
- remove secondary distractions when possible

### Feature page

- move from feature → outcome → example → proof
- show UI only when it clarifies value

### Comparison page

- acknowledge competitor strengths honestly
- win on 2-3 real differentiators
- avoid petty or unprovable claims

### About page

- explain why the product exists and why the team is credible
- connect the story back to customer benefit
- still include a clear CTA

## Voice and Tone

Decide the copy posture before drafting:

- formality: casual, professional-friendly, formal-enterprise
- personality: playful, serious, bold, understated, technical, accessible

Keep the system coherent:

- headlines can carry more personality
- body copy should stay concrete and readable
- CTAs should stay action-led

## Ethical Psychology and Mental Models

Use psychology to reduce friction and improve clarity, not to manipulate.

### Jobs to Be Done

Use when messaging is feature-heavy or mushy.

Frame the situation:

`When ___, you need ___ so you can ___.`

### Fogg Behavior Model

Use when users intend to act but do not complete.

- reduce friction
- make the next step obvious
- prompt at the right moment

### Hick's Law / choice reduction

Use when copy offers too many choices or too many CTAs.

- one primary CTA
- fewer options
- guided recommendation when appropriate

### Social proof

Use when trust is the blocker.

- put specific quotes, ratings, metrics, or logos near the claim and CTA

### Status-quo bias and switching cost

Use when adoption feels risky.

- emphasize easy start and first win
- explain migration or setup path clearly
- lower the fear of disruption

### Goal-gradient effect

Use for onboarding completion and setup progress.

- show momentum
- show what remains
- pull users toward the finish line honestly

### Peak-end rule

Use in onboarding, upgrade, checkout, and confirmation flows.

- create one clear aha moment
- end with confidence and an obvious next step

## Common Failure Modes

- wall of features without outcome translation
- platform fog and generic category language
- generic differentiation with no proof
- missing time-to-value
- trust claims with no support
- hidden or diluted CTA
- trying to sound expensive instead of being clear

Good product copy makes the value easier to understand and the next step easier to take.
